Consolidated archive of all bugs that have been fixed in `syara-x`. Supersedes
`tasks/04-12-2026-BUGS.md` (which carried these same entries with status:open
despite their fixes having shipped via the seven-batch plan in
`tasks/04-12-2026-BUGFIX-PLAN.md`).

Every fix is backed by a regression test (see plan doc for the per-batch
test list). Numbering is dense and historical — do not reuse numbers. The
next new bug should be **BUG-036**.

## Batch 1 (fixed) — Core data model + condition compilation

### BUG-001: Condition AST re-parsed on every scan call
- **File:** `syara/src/compiled_rules.rs:180`
- **Status:** fixed (Batch 1)
- **Description:** `evaluate_condition()` called `condition::parse(condition)` on every `scan()` invocation. Compiler parsed the AST to validate but discarded it.
- **Fix:** Compiler stores parsed `Expr` on `Rule`; `evaluate_condition` uses the stored AST — no re-parse.
- **Impact:** Wasted CPU on the hot path, multiplied by (rules × scans).

### BUG-006: No check for unconsumed tokens in condition parser
- **File:** `syara/src/condition.rs:301-305`
- **Status:** fixed (Batch 1)
- **Description:** After parsing an expression, parser did not verify next token was `Eof`. Input like `$s1 $s2` silently ignored `$s2`.
- **Fix:** EOF check added after `parse_expr()`.
- **Impact:** Silent rule misconfiguration.

### BUG-007: `is_identifier_needed` clones full HashMap
- **File:** `syara/src/compiled_rules.rs:360`
- **Status:** fixed (Batch 1)
- **Description:** `let mut test = current_matches.clone();` copied all match data per LLM rule per scan for hypothetical evaluation.
- **Fix:** Clone-free `evaluate_hypothetical` in `condition.rs`.
- **Impact:** Unnecessary allocations proportional to (pattern_count × llm_rule_count).

### BUG-017: `i64` with `-1` sentinel instead of `Option<usize>`
- **File:** `syara/src/models.rs:144-145`
- **Status:** fixed (Batch 1)
- **Description:** `start_pos` / `end_pos` used `i64` with `-1` sentinel — a Python-ism bypassing Rust's type system.
- **Fix:** Changed to `Option<usize>` for compile-time safety.
- **Impact:** API ergonomics.

## Batch 2 (fixed) — String matcher correctness + performance

### BUG-002: `fullword` modifier silently ignored
- **File:** `syara/src/engine/string_matcher.rs`
- **Status:** fixed (Batch 2)
- **Description:** `FullWord` modifier was parsed/cached but `compile()` never wrapped the pattern in `\b...\b`. Rules with `fullword` matched as if the modifier were absent.
- **Fix:** `compile()` wraps pattern in `\b...\b` when FullWord is set.
- **Impact:** Silent false positives.

### BUG-009: `Regex::clone()` on every match call
- **File:** `syara/src/engine/string_matcher.rs:76`
- **Status:** fixed (Batch 2)
- **Description:** Compiled regex cloned to work around borrow conflict. `Regex::clone()` copies NFA/DFA internals.
- **Fix:** Split `compile()` to avoid the clone.
- **Impact:** Allocation on every string match call.

### BUG-010: Double HashMap lookup in `StringMatcher::compile()`
- **File:** `syara/src/engine/string_matcher.rs:37-60`
- **Status:** fixed (Batch 2)
- **Description:** `contains_key` → index → `insert` → index: four lookups where one `entry()` suffices.
- **Fix:** Use `self.cache.entry(key)`.
- **Impact:** Minor inefficiency on hot path.

### BUG-025: Wide regex not cached
- **File:** `syara/src/engine/string_matcher.rs:103-130`
- **Status:** fixed (Batch 2)
- **Description:** `match_wide` built and compiled a new `Regex` on every call.
- **Fix:** Wide regexes cached with `:wide_regex`/`:wide_literal` suffix keys.

### BUG-026: Wide regex positions refer to stripped text
- **File:** `syara/src/engine/string_matcher.rs:110-114`
- **Status:** fixed (Batch 2)
- **Description:** Reported positions referred to null-stripped text, not original.
- **Fix:** Stripped-to-original position map.

## Batch 3 (fixed) — Parser performance + correctness

### BUG-004: Parser regexes recompiled on every call
- **File:** `syara/src/parser.rs` (multiple lines)
- **Status:** fixed (Batch 3)
- **Description:** Every section parser called `Regex::new().unwrap()` inline — ~10+ compilations per rule.
- **Fix:** 7 `LazyLock<Regex>` statics across `parser/mod.rs` + `parser/sections.rs`.
- **Impact:** CPU cost when parsing large rule files.

### BUG-005: O(n²) string rebuild in `split_rules`
- **File:** `syara/src/parser.rs:150-161`
- **Status:** fixed (Batch 3)
- **Description:** `chars[i..].iter().collect::<String>()` rebuilt the tail on every iteration of the outer loop.
- **Fix:** Reworked to use byte offsets on `&str` via `content.as_bytes()`.
- **Impact:** Quadratic time on files with many rules.

### BUG-020: String regex doesn't handle escaped quotes
- **File:** `syara/src/parser.rs:342`
- **Status:** fixed (Batch 3)
- **Description:** `[^"]*` stopped at the first `"` — `$s = "say \"hello\""` mis-parsed.
- **Fix:** Changed to `(?:[^"\\]|\\.)*`; added `unescape_string()` for `\"`, `\\`, `\n`, `\t`, `\r`.

### BUG-021: Missing condition silently returns empty string
- **File:** `syara/src/parser.rs:586`
- **Status:** fixed (Batch 3)
- **Description:** Rule without a condition section got `condition: ""` evaluating to `false`. Typos like `conditon:` silently produced never-matching rules.
- **Fix:** `parse_rule_block` returns error when patterns exist but condition is missing/empty.

## Batch 4 (fixed) — phash / audio safety + `scan_file`

### BUG-003: Bit shift overflow in `ImageHashMatcher` for `hash_size > 8`
- **File:** `syara/src/engine/phash_matcher.rs:97-103`
- **Status:** fixed (Batch 4)
- **Description:** `1u64 << (row * hash_size + col)` exceeded 63 bits when `hash_size > 8`. Panicked in debug, wrapped in release.
- **Fix:** `compute_hash` returns `Err` for `hash_size > MAX_HASH_SIZE` (8).

### BUG-008: `scan_file` skips string patterns in mixed phash+string rules
- **File:** `syara/src/compiled_rules.rs:60-66`
- **Status:** fixed (Batch 4)
- **Description:** `scan_file` passed `text = ""` to the engine. Mixed phash+string rules (`$phash1 and $str1`) silently never matched.
- **Fix:** `scan_file` reads content via `String::from_utf8_lossy(std::fs::read(path))` for string matchers.

### BUG-014: WAV parser ignores channel count
- **File:** `syara/src/engine/phash_matcher.rs:141-166`
- **Status:** fixed (Batch 4)
- **Description:** `n_frames = chunk_len / sample_width.max(1)` didn't account for channels. Stereo files reported double the frame count.
- **Fix:** Reads `n_channels` from fmt offset 2; divides by `(n_channels * sample_width)`; sampling loop seeks by `frame_size`.

### BUG-015: Unchecked WAV `chunk_len` allocation
- **File:** `syara/src/engine/phash_matcher.rs:154`
- **Status:** fixed (Batch 4)
- **Description:** `vec![0u8; chunk_len as usize]` allocated directly from an untrusted WAV header field. DoS via crafted file with 4GB `chunk_len`.
- **Fix:** `MAX_CHUNK_ALLOC = 256MB`; fmt and data chunks rejected if `chunk_len > MAX_CHUNK_ALLOC`.

### BUG-029: WAV parser doesn't handle RIFF odd-chunk padding
- **File:** `syara/src/engine/phash_matcher.rs:169`
- **Status:** fixed (Batch 4)
- **Description:** RIFF spec requires even-byte chunk boundaries. Parser seeked by exact `chunk_len`, leaving alignment drift on odd-length chunks.
- **Fix:** Unknown/fmt chunks with odd `chunk_len` seek past 1 padding byte per RIFF spec.
- **Note:** This is the *original* BUG-029. See **BUG-035** for the later LLM-truncation bug that was briefly mis-labelled with the same number.

## Batch 5 (fixed) — HTTP clients + shared helpers

### BUG-011: No HTTP timeouts on blocking clients
- **File:** `syara/src/engine/semantic_matcher.rs:110`, `classifier.rs:76`, `llm_evaluator.rs:71`
- **Status:** fixed (Batch 5)
- **Description:** `reqwest::blocking::Client::new()` had no timeout. Hanging server blocked the scanner thread.
- **Fix:** 10s connect + 30s read timeouts on all HTTP clients (LLM doubled to 20s/60s later during the HTTP LLM refactor).

### BUG-012: Duplicated `embed()` implementation
- **File:** `syara/src/engine/classifier.rs` vs `semantic_matcher.rs`
- **Status:** fixed (Batch 5)
- **Description:** `HttpEmbeddingClassifier::embed()` was character-for-character identical to `HttpEmbeddingMatcher::embed()` — divergence risk.
- **Fix:** Shared `HttpEmbedder` in `engine/mod.rs`; both callers delegate.

### BUG-013: LLM prompt injection surface
- **File:** `syara/src/engine/llm_evaluator.rs:75-83`
- **Status:** fixed (Batch 5)
- **Description:** Both `pattern` and `input_text` interpolated directly into the LLM prompt via `format!()`. Malicious input could manipulate the response.
- **Fix:** `build_prompt` uses XML delimiters `<pattern>` and `<input>` around untrusted content; injection surface documented in doc comments.

### BUG-033: Pattern embeddings recomputed on every `match_chunks` call
- **File:** `syara/src/engine/semantic_matcher.rs:42`
- **Status:** fixed (Batch 5)
- **Description:** `self.embed(&rule.pattern)` called per invocation. Repeated scans with the same rule redundantly embedded the pattern.
- **Fix:** `HttpEmbedder` includes a `Mutex<HashMap>` embedding cache.

### BUG-034: `SetExpr::Them` returns keys in arbitrary HashMap order

## 2026-04-19 — HTTP LLM refactor closeout

### BUG-035: Reasoning-model truncation silently read as "Ambiguous"
- **File:** `syara/src/engine/llm_evaluator.rs`
- **Status:** fixed (2026-04-19)
- **Discovered during:** `integration_real_openai_chat` test against LM Studio's `qwen/qwen3.6-35b-a3b`.
- **Description:** Reasoning models (Qwen3.x, DeepSeek-R1, GPT-OSS) spend thousands of tokens on internal `reasoning_content` before emitting the final YES/NO in `content`. With the old default `max_tokens = 128`, `content` came back empty with `finish_reason = "length"`, and the evaluator fell through `parse_response` to `"Ambiguous LLM response:"` — silent false negative.
- **Fix:**
  1. `DEFAULT_MAX_TOKENS` 128 → **8192** (reasoning headroom; non-reasoning models still stop early via `finish_reason=stop`).
  2. Extracted pure helper `extract_openai_content(json) -> Result<String, …>` for testability.
  3. Empty `content` with `finish_reason=length` now returns `SyaraError::LlmError` pointing at `.max_tokens(…)` instead of falling through.
  4. Non-empty content with `finish_reason=length` still accepted (first YES/NO token usually already present).
  5. Five new unit tests: ok / empty+length / empty+stop / non-empty+length / missing choices.
- **Numbering note:** Session notes labelled this BUG-029. Renumbered to **BUG-035** here to break the collision with the WAV RIFF odd-chunk-padding bug (also BUG-029, fixed in Batch 4). Source comments, `todo.md` references, and memory files are being updated to match.
- **Verified:** clippy `--features llm -D warnings` clean; qwen3.6-35b now returns clean YES in ~10s; gemma-3-4b regression still 0.4s.
